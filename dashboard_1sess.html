1. Project Context, Scope, and Objectives
Context and problem statement

Context. The app is a social video discussion platform where users reply to each other with short videos. New users were dropped directly into a generic, unpersonalized feed.

Observed issues.

  • First-session engagement plateauing (~48% meaningful engagement in session 1).
  • 7-day retention (R7) at ~22% for new users.
  • No lightweight way to capture user interests on the first session.

Strategic question. Can we use declared interests during onboarding to make the first experience feel “instantly relevant” and improve early-life retention?

Scope of this sprint.

  • In scope: New-user onboarding flow, interest capture UI, first-session feed seeding, experimentation & analysis.
  • Out of scope: Pricing/monetization changes, long-term content supply strategy, brand marketing, and creator incentives.

Primary objectives.

  • Increase first-session engagement with minimal added friction.
  • Improve 7-day retention for new users.
  • Protect key guardrails (latency, crash rate, support tickets, spend/DAU).

Hypothesis and success criteria

Hypothesis. Asking new users to pick at least three interests and seeding their first feed from those interests will:

  • Increase perceived relevance of content in session 1.
  • Accelerate time to first meaningful action (e.g., like, reply).
  • Translate into higher 7-day retention without harming reliability or monetization.

2. Approach to Identify Opportunities
Experiment design and overall approach

Design: controlled experiment.

  • Randomized controlled test at the user_id level.
  • 50/50 split between Control (A) and Test (B).
  • Stratified by platform (iOS/Android) and geo (US / non-US).
  • Fixed 14-day observation window from signup.

Assignment and logging.

  • Experiment key: exp_onb_pick3_v1.
  • Feature flag: onb.pick3.enabled (kill switch present).
  • Exclusions: returning users, employees, flagged bots, age < 13.

Analysis methodology.

  • Primary metric: 7-day retention (R7); secondary: first-session engagement, onboarding completion.
  • Difference-in-proportions with pooled standard error.
  • CUPED adjustment using baseline activity to reduce variance.
  • MDE for R7: 3.0 pp at 95% power, α = 0.05.
  • Stopping rule: fixed horizon + sequential monitoring with alpha spending; no early stop.
Pre-AA test (n ≈ 4,800) showed no metric drift (all p > 0.3), and no sample ratio mismatch was detected in the live test (χ² p = 0.41; split within ±0.8%).

2. Approach to Identify Opportunities
Solution design: interest picker and feed seeding

Micro-onboarding “Pick 3 interests”.

  • Inserted an interest selection step into the onboarding funnel for new users.
  • Required users to pick ≥3 interests from a curated taxonomy (e.g., art, tech, travel).
  • Interests stored as non-PII IDs with a 180-day retention policy.

User experience principles.

  • Fast, tappable grid UI optimized for thumb reach.
  • Clear progress affordance (“Pick at least 3 to continue”).
  • Copy emphasizing personalization (“We’ll tailor your first feed to what you care about”).

Personalized first feed.

  • For each selected interest, pulled top-N content items from a relevance-ranked index.
  • Backfilled with trending content to mitigate long-tail sparsity.
  • Prefetched content for selected interests to avoid first-session latency spikes.
  • All logic guarded by the onb.pick3.enabled feature flag.

Key artifacts.

  • Jira ticket: ONB-241 (micro-onboarding & feed seeding).
  • Server commit: 6b91e3a; client commit: c21ad7f.
  • Dashboards: Metabase R7 Cohorts v1, Datadog FE Latency p95.

3. Key Findings and Recommendations
Executive summary – key findings

Overall conclusion. The “Pick 3 interests” + personalized first feed experience outperformed control on both engagement and 7-day retention, with guardrails green.

First-Session Engagement
+9 pp
Baseline 48% → 57% in test (B–A).
7-Day Retention (R7)
+6 pp
22% → 28%; 95% CI [+3.4, +8.6]; p = 0.004.
Onboarding Completion
82%
Up from 75% baseline (+7 pp) despite added step.
  • Time-to-first-like (median) improved from 2.8 to 2.4 minutes (−0.4).
  • Crash rate and latency were effectively unchanged; monetization metrics (spend/DAU) remained within ±1% band.
  • Impact was strongest for organic US users but positive across all major segments.
Recommendation: Promote the interest picker and personalized feed to the default experience for all new users, and iterate further using segments where lift was highest.

4. Targeted Impact and Other Benefits
Guardrails and reliability remained within SLOs

Technical guardrails.

  • Crash rate: 0.44% (A) vs 0.42% (B) — Δ −0.02 pp (no concern).
  • FE p95 latency: +8 ms vs baseline; alert threshold +20 ms.
  • Data freshness SLO: 95th percentile ETL lag 11 minutes (SLO < 15m).

Business guardrails.

  • Spend/DAU: no impact outside ±1% band.
  • Support tickets: 12 → 11 per week (no increase; slight improvement).

Data quality checks.

  • Event coverage ≥ 99.2% for: onboarding_started, interests_selected, feed_viewed, first_like, retained_7d.
  • Late-arriving events < 1.2%, handled via 24-hour watermark.
  • Timezone normalized to UTC; device clock skew mitigated via server timestamps.
  • Bot/fraud filtered using rules engine v3.
With guardrails and data quality checks in place, we can attribute observed lifts to the new onboarding + feed experience with high confidence.

5. Enabling Implementation
Rollout approach and risk mitigation

Rollout plan.

  • Staged feature-flag rollout: 1% → 10% → 25% → 50% → 100% with health checks at each step.
  • Monitors in Datadog for:
    • FE p95 latency, error rate, crash rate.
    • Experiment assignment consistency.
  • Runbook / rollback: onb.pick3.enabled=false + revert PR#1284 if any guardrail breached.

Stakeholder communication.

  • Release notes and FAQs for CX/support teams.
  • GTM playbook to update UA creatives with “tailored first feed” messaging.

Key risks and mitigations.

  • Cold-start misclassification.
    Mitigation: default safe categories + trending backfill if interests sparse.
  • Long-tail content sparsity.
    Mitigation: cap percentage of long-tail items; blend with popular adjacent topics.
  • Latency spike on first session.
    Mitigation: prefetch top-N content per interest prior to first feed render.
Given strong results and low residual risk, the recommendation is to fully adopt the “Pick 3 interests” onboarding as the default for all new users.

5. Enabling Implementation
Next steps and roadmap (Q4 2024)

We recommend three follow-on initiatives to build on the success of this sprint.

  1. Smart Interest Expansion (ONB-274).
    Use behavior in days 1–3 to suggest adjacent topics and help users refine their interest graph.
  2. Feed Rebalance v2 (FEED-198).
    Tune weights for implicit signals (watch time, replies, skips) and decay older interests over time.
  3. Live Retention Cohorts (DATA-311).
    Build real-time R7 / R30 dashboards with anomaly alerts to monitor downstream impact of future changes.
Together, these initiatives create a roadmap from “static declared preferences” to a dynamic, behavior-informed understanding of user interests over time.

Appendix – Data model and analysis SQL (excerpt)

Event schema (simplified).

TABLE user_events ( event_time_utc TIMESTAMP, user_id STRING, event_name STRING, -- e.g. 'interests_selected' properties JSON, -- { "count": 3, "ids": ["art","tech","travel"] } platform STRING, -- 'ios' | 'android' | 'web' acquisition STRING, -- 'organic' | 'paid' experiment_key STRING, -- 'exp_onb_pick3_v1' variant STRING -- 'A' | 'B' );

Variant-level R7 summary (CUPED-adjusted).

WITH agg AS ( SELECT variant, COUNT(DISTINCT user_id) AS users, AVG(CASE WHEN retained_7d THEN 1 ELSE 0 END) AS r7 FROM experiment_sessions WHERE experiment_id = 'exp_onb_pick3_v1' AND signup_at BETWEEN '2024-09-29' AND '2024-10-12' GROUP BY 1 ) SELECT *, (MAX(r7) - MIN(r7)) * 100 AS lift_pp FROM agg;

Segment lift by platform × acquisition.

SELECT platform, acquisition, AVG(retained_7d) FILTER (WHERE variant='B') - AVG(retained_7d) FILTER (WHERE variant='A') AS lift FROM experiment_sessions GROUP BY 1,2 ORDER BY 3 DESC;
